你好，OpenClaw。
接下来需要对系统中的模型配置模块进行一次架构级重构。目标是将系统中的模型能力从具体业务逻辑中解耦，并建立统一的模型管理机制，使未来可以在不同任务中灵活调用不同模型。
请按照以下执行计划对系统进行修改。
一、建立独立的模型管理模块
在系统中新增一个独立的模型管理模块，用于统一管理所有 AI 模型资源。
该模块必须与具体业务逻辑解耦，作为系统的基础能力层存在。
模型需要支持两种类型：
1）在线模型
在线模型通过远程 API 调用，需要支持以下通用配置能力：
- 模型名称
- 服务提供商标识
- API 接口地址
- 认证信息（如 API Key 等）
- 可扩展的请求参数配置
- 可用性检测机制
系统需要提供完整的管理能力，包括：
- 新建模型配置
- 修改模型配置
- 删除模型配置
- 状态检测与状态展示
2）本地模型
本地模型运行在用户本地环境，需要支持以下能力：
- 本地模型标识
- 本地路径或运行方式
- 安装脚本或安装指引
- 环境检测机制
- 模型可用性检测
- 状态展示
系统需要能够检测本地模型是否存在、是否可运行，并向用户展示当前状态。
二、建立模型标签系统
所有模型必须支持标签机制，用于表达模型能力或适用领域。
标签是可扩展的数据结构，一个模型可以拥有多个标签。
标签用于在系统中对模型进行能力分类，并用于后续任务选择模型时进行筛选。
系统在 UI 中展示模型时，应当将标签与模型名称进行组合展示，以帮助用户快速理解模型能力，但不暴露底层配置细节。
三、实现模型与任务解耦
系统中的任务或业务功能不得直接依赖具体模型实现。
任务只能引用模型管理模块中已注册的模型。
每个任务需要支持配置一个或多个模型。
任务运行时由系统根据任务配置调用对应模型。
任务层只关注所需能力，不关心模型来源或实现方式。
四、建立通用调用机制
系统需要设计统一的模型调用接口，使在线模型和本地模型可以通过统一方式被调用。
调用层需要屏蔽模型类型差异，使任务层无需关心模型是在线还是本地。
五、重构目标
本次修改的目标是建立一个可扩展的模型管理体系，使系统能够：
- 统一管理所有模型资源
- 支持在线模型与本地模型
- 支持模型能力标签分类
- 支持任务灵活选择多个模型
- 支持未来新增模型类型而无需修改业务逻辑
请在现有系统结构基础上完成该模块重构，并确保模块之间职责清晰、结构可扩展。
